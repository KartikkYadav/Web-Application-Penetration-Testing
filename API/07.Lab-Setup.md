```

```

```

```

```
curl -L -o /tmp/crapi.zip https://github.com/OWASP/crAPI/archive/refs/heads/main.zip
```

```
unzip /tmp/crapi.zip
```

```
cd crAPI-main/deploy/docker
```

```
docker-compose pull
```


# 3. What vapi Is

vapi is a **minimal intentionally vulnerable REST API**.

It is lightweight compared to **crAPI** and ideal for:

- API enumeration practice
- Injection testing
- Authentication bypass testing
- IDOR discovery
- Basic fuzzing

It is good as a **starter lab before crAPI** in your API pentesting workflow.

---

# 4. Initial Enumeration

## Step 1 – Directory Enumeration

    ffuf -u http://192.168.1.28/FUZZ -w /usr/share/wordlists/dirb/common.txt

Also try:

    ffuf -u http://192.168.1.28/vapi/FUZZ -w wordlist.txt

---

## Step 2 – HTTP Method Testing

    curl -X OPTIONS http://192.168.1.28/vapi/api1 -i

Test methods:

- GET
- POST
- PUT
- DELETE

Sometimes insecure methods are enabled.

---

## Step 3 – Parameter Discovery

    ffuf -u "http://192.168.1.28/vapi/api1?FUZZ=test" -w param.txt

Look for:

- Hidden parameters
- Debug flags
- Admin flags

---

# 5. Common Vulnerabilities in vapi

---

## 1. SQL Injection

Test payloads:

    ' OR 1=1--

    " OR "1"="1

Use sqlmap:

    sqlmap -u "http://192.168.1.28/vapi/api1?id=1" --batch --dbs

---

## 2. Authentication Bypass

Try payloads:

    admin'--

    admin" OR "1"="1

Check for:

- Hardcoded credentials
- Missing password validation
- JWT manipulation (if present)

---

## 3. IDOR (Broken Object Level Authorization)

Check for:

- Horizontal privilege escalation
- Access to other users' data

Example:

    GET /vapi/api1/users/1001

Change ID:

    GET /vapi/api1/users/1002

---

## 4. Input Validation Issues

Test:

- Long strings
- Special characters
- JSON tampering
- Negative values
- Large numeric values

---

## 5. Mass Assignment

If JSON body is accepted:

    {
      "username": "rahul",
      "role": "admin"
    }

Try adding fields:

- isAdmin
- role
- balance
- user_id

---

# 6. Burp Testing Flow (Recommended)

1. Proxy request through Burp
2. Send request to Repeater
3. Modify request

Test changes:

- ID values
- HTTP method
- Headers
- JSON body

4. Observe response differences

# 3.VAmPI – Lab Setup & Testing Guide

Repository: https://github.com/erev0s/VAmPI

VAmPI is a **Vulnerable API built in Flask**.

It is intentionally designed to demonstrate **OWASP API Security Top 10 vulnerabilities** in a clean REST architecture.

---

# 1. Installation Methods

You can use either method.

## Option 1 – Run Prebuilt Image

    docker run -p 5000:5000 erev0s/vampi:latest

---

## Option 2 – Clone + Docker Compose

    cd /opt
    git clone https://github.com/erev0s/VAmPI.git
    cd /opt/VAmPI
    docker-compose up -d

Verify services:

    docker ps
    netstat -nltup

---

# 2. Access URLs

If your IP is:

    192.168.1.28

| URL | Purpose |
|-----|--------|
| http://192.168.1.28:5001 | API Root |
| http://192.168.1.28:5001/ui/ | Swagger UI |

⚠️ Swagger UI is extremely useful for reconnaissance.

---

# 3. Architecture Overview

VAmPI characteristics:

- REST API
- Flask-based
- JWT Authentication
- Single container (simpler than crAPI)
- SQLite/Postgres depending on configuration

Unlike crAPI (microservices), **VAmPI is monolithic**, which makes it ideal for learning core API vulnerabilities.

---

# 4. Initial Recon Strategy

## Step 1 – Swagger Enumeration

Visit:

    http://192.168.1.28:5001/ui/

Check:

- All endpoints
- Required parameters
- Authentication requirements
- Response schema

Swagger provides a **complete attack surface overview**.

---

## Step 2 – Identify Authentication Flow

Common endpoints:

    POST /users/v1/login
    POST /users/v1/register

Capture the **JWT token** after login.

Inspect token parts:

- Header
- Payload
- Signature

---

# 5. Common Vulnerabilities in VAmPI

VAmPI intentionally includes multiple **OWASP API vulnerabilities**.

---

## 1. Broken Object Level Authorization (BOLA)

Example:

    GET /users/v1/users/1
    GET /users/v1/users/2

Change the **user ID** and check if other users' data is accessible.

---

## 2. JWT Weakness

Check for:

- Hardcoded secret keys
- Weak signing algorithm
- Role tampering

Try:

- Modify role: admin
- Change user_id
- Re-sign token if secret is discovered

---

## 3. Mass Assignment

Example JSON request:

    {
        "username": "rahul",
        "role": "admin"
    }

Try injecting additional fields:

- isAdmin
- role
- balance
- user_id

---

## 4. Excessive Data Exposure

Example response:

    {
        "id": 1,
        "username": "admin",
        "password": "hashedvalue",
        "secret_key": "..."
    }

Look for:

- Internal fields
- Debug information
- Unnecessary sensitive data

---

## 5. Rate Limiting Issues

Brute force login:

    ffuf -X POST \
    -d '{"username":"admin","password":"FUZZ"}' \
    -u http://192.168.1.28:5001/users/v1/login \
    -w rockyou.txt

Check:

- Account lockout
- Rate limiting
- Response differences

---

## 6. Injection Testing

Test for:

- SQL Injection
- Command Injection in search parameters
- JSON Injection

---

# 6. Recommended Pentest Flow

1. Swagger reconnaissance
2. Register account
3. Capture JWT token
4. Test:

   - BOLA
   - Role tampering
   - ID enumeration

5. Test mass assignment
6. Test rate limiting
7. Test injection
8. Analyze error handling

# 4. Damn Vulnerable Bank – Lab Setup & API Pentesting Guide

Repository: https://github.com/rewanthtammana/Damn-Vulnerable-Bank

Backend Source:  
https://github.com/rewanthtammana/Damn-Vulnerable-Bank/tree/master/BackendServer

Damn Vulnerable Bank (DVB) is a **deliberately vulnerable banking API** designed for practicing **API security testing and business logic attacks**.

It simulates a **financial backend system** where attackers can exploit **authentication flaws, BOLA, transaction manipulation, race conditions, and injection vulnerabilities**.

---

# 1. Installation (Backend API)

Clone the repository and start the backend server using Docker.

    git clone https://github.com/rewanthtammana/Damn-Vulnerable-Bank.git
    cd /opt/Damn-Vulnerable-Bank/BackendServer
    docker-compose up -d

Verify the containers:

    docker ps
    netstat -nltup

---

# 2. Access URLs

If your machine IP is:

    192.168.1.28

| URL | Purpose |
|-----|--------|
| http://192.168.1.28:3000 | Application root |
| http://192.168.1.28:3000/api | API root |
| http://192.168.1.28:3000/api/health/check | Health endpoint |

The health endpoint can sometimes expose **debug or environment information**.

---

# 3. Architecture Overview

DVB simulates a **banking backend API**.

Typical stack includes:

- Node.js / Express backend
- MongoDB or relational database
- REST APIs
- JWT-based authentication
- Financial transaction logic

Unlike VAmPI, this lab focuses heavily on:

- **Business logic flaws**
- **Transaction manipulation**
- **Authorization bypass**
- **Financial integrity issues**

---

# 4. Initial Recon Strategy

## Step 1 – API Enumeration

Use directory fuzzing to discover endpoints.

    -w /usr/share/wordlists/dirb/common.txt

Look for endpoints such as:

- /users
- /accounts
- /transfer
- /transactions
- /admin
- /auth

---

## Step 2 – Check Allowed HTTP Methods

    curl -X OPTIONS http://192.168.1.28:3000/api/ -i

Test additional methods:

- PUT
- DELETE
- PATCH
- Unexpected HTTP methods

These sometimes expose **hidden functionality or misconfigurations**.

---

# 5. Common Vulnerability Categories in DVB

DVB intentionally includes several **real-world API security issues**.

---

## 1. Broken Authentication

Test for:

- Weak JWT secret
- Expired token reuse
- Role tampering
- Missing signature validation

Inspect JWT tokens:

    jwt_tool <token>

Try modifying payload:

    {
        "role": "admin",
        "balance": 999999
    }

---

## 2. Broken Object Level Authorization (BOLA)

Example requests:

    GET /api/accounts/1001
    GET /api/accounts/1002

Modify the **account ID** to access other users' financial data.

---

## 3. Business Logic Flaws (Most Important)

### Negative Transfer

Try sending a negative amount:

    {
        "toAccount": "1002",
        "amount": -5000
    }

Check if the balance **increases instead of decreasing**.

---

### Large Amount Transfer

Test for:

- Integer overflow
- Floating point rounding errors
- Decimal manipulation

---

### Self Transfer

Transfer from the same account.

Example:

Account A → Account A

Check for:

- Duplicate credits
- Transaction duplication

---

### Race Condition

Send multiple requests simultaneously.

Example:

    seq 1 20 | xargs -I{} -P20 curl ...

Check if:

- Double spending occurs
- Balance validation fails under concurrency

---

## 4. Mass Assignment

Try injecting additional fields:

    {
        "username": "rahul",
        "role": "admin",
        "balance": 1000000
    }

Check if the API accepts **unauthorized parameters**.

---

## 5. Injection Testing

Test for:

- SQL Injection
- NoSQL Injection (MongoDB)
- JSON Injection
- Command injection in logs

Example NoSQL injection:

    {
        "username": {"$ne": null},
        "password": {"$ne": null}
    }

---

## 6. ID Enumeration

Look for predictable identifiers:

- Account numbers
- Transaction IDs
- User IDs

Increment or decrement IDs to access other records.

---

# 6. Recommended Pentest Flow

Since DVB simulates a **banking system**, follow this workflow.

1. Register a new user
2. Login and capture JWT token
3. Identify your account ID

Then test:

- BOLA on account endpoints
- Transaction manipulation
- Business logic flaws

Continue with:

5. Test admin endpoints  
6. Test race conditions  
7. Check for sensitive data exposure
