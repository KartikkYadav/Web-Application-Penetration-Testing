# OWASP API Security Top 10

The **OWASP API Security Top 10** is published by OWASP (Open Web Application Security Project).  
It highlights the most critical security risks affecting APIs.

- The latest stable release is:
  - **OWASP API Security Top 10 (2023)**

This list focuses specifically on **API-related risks**, not traditional web application vulnerabilities.

---

# OWASP API Security Top 10 (2023)

---

# API1: Broken Object Level Authorization (BOLA)

## What It Is

APIs expose objects such as **users, orders, and accounts**.  
If the API does not properly verify object ownership, attackers can access other users' data by manipulating object identifiers.

## Example

```
GET /api/users/124
```

If **user 123** can access **user 124's data** by changing the ID → **BOLA vulnerability**.

## Impact

- Data leakage
- Account takeover
- Financial fraud

## Prevention

- Always validate ownership **server-side**
- Do not rely on **client-side checks**

---

# API2: Broken Authentication

## What It Is

Authentication mechanisms are implemented incorrectly, allowing attackers to compromise tokens or credentials.

## Common Issues

- Weak JWT secrets
- Missing token expiration
- Brute force without rate limiting
- Accepting unsigned JWT (`alg: none`)

## Impact

- Account takeover
- Privilege escalation

## Prevention

- Use strong token secrets
- Enforce token expiration
- Implement MFA
- Rate limit login endpoints

---

# API3: Broken Object Property Level Authorization

## What It Is

Also called **Mass Assignment vulnerability**.

APIs allow modification of properties that should not be controlled by the user.

## Example

```
{
  "username": "john",
  "role": "admin"
}
```

If the **role field** can be modified by the user → **privilege escalation**.

## Impact

- Privilege escalation
- Data corruption

## Prevention

- Whitelist allowed fields
- Use DTOs (Data Transfer Objects)
- Do not auto-bind entire objects

---

# API4: Unrestricted Resource Consumption

## What It Is

APIs do not limit resource usage, allowing abuse.

## Examples

- No rate limiting
- Large payload uploads
- Expensive queries
- Infinite pagination

## Impact

- Denial of Service (DoS)
- Increased infrastructure cost

## Prevention

- Rate limiting
- Throttling
- Payload size limits
- Query depth limiting (GraphQL)

---

# API5: Broken Function Level Authorization

## What It Is

Improper authorization checks on sensitive functions.

## Example

```
POST /api/admin/deleteUser
```

If a normal user can access an **admin endpoint**, it leads to **privilege escalation**.

## Impact

- Unauthorized actions
- Account manipulation
- Data loss

## Prevention

- Implement role-based access control (RBAC)
- Validate permissions server-side
- Separate admin and user endpoints

---

# API6: Unrestricted Access to Sensitive Business Flows

## What It Is

Business logic abuse due to lack of controls.

## Examples

- Unlimited OTP requests
- Coupon abuse
- Automated ticket booking
- Brute-forcing verification codes

## Impact

- Fraud
- Financial loss
- Service abuse

## Prevention

- Rate limiting per action
- CAPTCHA
- Anti-automation controls
- Behavioral monitoring

---

# API7: Server-Side Request Forgery (SSRF)

## What It Is

The API fetches remote resources without validation, allowing attackers to make the server send requests to internal systems.

## Example

```
POST /api/fetch
{
  "url": "http://internal-service/admin"
}
```

## Impact

- Access to internal services
- Cloud metadata exposure
- Internal network scanning

## Prevention

- Validate and sanitize URLs
- Use allowlists for outbound requests
- Block internal IP ranges
- Disable unnecessary network access

---

# API8: Security Misconfiguration

## What It Is

Improperly configured API infrastructure or services.

## Examples

- Debug endpoints enabled
- Default credentials
- Misconfigured CORS
- Unnecessary HTTP methods enabled

## Impact

- Information disclosure
- Unauthorized access
- System compromise

## Prevention

- Disable unnecessary services
- Secure configuration management
- Regular security audits
- Harden API gateways

---

# API9: Improper Inventory Management

## What It Is

Organizations do not properly track APIs, leading to **shadow or outdated APIs**.

## Examples

- Deprecated API versions still active
- Undocumented endpoints
- Test endpoints exposed in production

## Impact

- Attackers find forgotten endpoints
- Sensitive data exposure
- Security policy bypass

## Prevention

- Maintain API inventory
- Deprecate unused APIs
- Document all endpoints
- Monitor API usage

---

# API10: Unsafe Consumption of APIs

## What It Is

Applications trust **third-party APIs** without proper validation.

## Examples

- Trusting external API responses blindly
- No validation of third-party data
- Dependency on insecure external services

## Impact

- Data corruption
- Injection attacks
- Supply chain attacks

## Prevention

- Validate external API responses
- Implement schema validation
- Use API gateways
- Monitor third-party integrations
