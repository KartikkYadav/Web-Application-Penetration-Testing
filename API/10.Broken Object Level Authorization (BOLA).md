# 🔐 Broken Object Level Authorization (BOLA)

---

# 📌 Overview

BOLA occurs when:

- The application **does NOT verify object ownership**
- User can **manipulate object IDs**
- Access **other users' data**

👉 Most common in APIs (especially REST)

---

# 🧠 Core Concept

Instead of checking:

    "Does this user own this object?"

The system only checks:

    "Is the user authenticated?"

❌ This leads to **IDOR-style attacks in APIs**

---

# ⚠️ Vulnerability Description

- API exposes object identifiers:
  - `vehicle_id`
  - `order_id`
  - `report_id`

- Backend fails to validate ownership

👉 Result:

- Authenticated user can access:
  - Other users' data
  - Sensitive resources

---

# 🎯 Affected Endpoints

---

## 🚗 1. Vehicle Location API

### Endpoint

    GET /identity/api/v2/vehicle/{vehicle_id}/location

---

### Example Request

    GET /identity/api/v2/vehicle/4bae9968-ec7f-4de3-a3a0-ba1b2ab5e5e5/location HTTP/1.1
    Host: 192.168.1.28:8888
    Authorization: Bearer <user_token>

---

### 🧪 Exploitation Steps

1. Login as User A  
2. Capture request  
3. Replace `vehicle_id` with User B's ID  

    /identity/api/v2/vehicle/<victim_id>/location

4. Send request  

---

### ❌ Result

- Returns **location of another user's vehicle**

---

### ✅ Expected

    403 Forbidden
    OR
    Access Denied

---

## 📦 2. Orders API

### Endpoint

---

### Example

    GET /workshop/api/shop/orders/1
    GET /workshop/api/shop/orders/9

---

### 🧪 Exploitation Steps

1. Login as User A  
2. Capture request  

    GET /orders/1

3. Modify ID  

    GET /orders/9

4. Send request  

---

### ❌ Result

- Returns **another user's order details**

---

## 🛠️ 3. Mechanic Reports API

### Endpoint

    GET /workshop/api/mechanic/mechanic_report?report_id=2

---

### ❌ Result

- Access to **other users' reports**

# 🧨 Root Cause

Backend does NOT enforce:

    Object ownership validation

---

## ❌ Current Logic

    if user is authenticated:
        allow access

---

## ✅ Required Logic

    if object.user_id == current_user.id:
        allow
    else:
        deny

---

# 🛡️ Remediation

---

## 1. Enforce Object-Level Authorization

    if order.user_id != current_user.id:
        return 403

---

## 2. Use Access Control Middleware

- Centralize authorization checks
- Avoid scattered logic

---

## 3. Use Indirect Object References

- Avoid exposing:
  - Sequential IDs
- Use:
  - UUIDs
  - Secure mappings

---

## 4. Apply Authorization at DB Layer

    SELECT * FROM orders
    WHERE order_id = ?
    AND user_id = current_user_id;

---

## 5. Logging & Monitoring

Detect:

- ID enumeration attempts
- Access to foreign objects

---

# 🔍 Testing Tips (Pentesting)

- Look for:
  - ID in URL (`/user/123`)
  - ID in params (`?id=123`)
  - JSON body IDs

---

## 🔁 Change IDs

- Increment IDs
- Use random UUIDs
- Test with other users

---

## 🔥 Indicators of BOLA

- No 403 response
- Data leakage
- Different user data visible

---
